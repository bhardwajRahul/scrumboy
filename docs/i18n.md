# Frontend i18n architecture

Contributor-facing overview of Scrumboy’s SPA localization. **Rules and constraints** (what may change in a localization PR) live in [`AGENTS.md`](../AGENTS.md)—this page describes architecture and change gates only.

## Catalogs

- **`en.json`** is the source catalog.
- Every key must exist in **every** locale file under [`internal/httpapi/web/modules/i18n/locales/`](../internal/httpapi/web/modules/i18n/locales/) (all public locales plus `pseudo`).
- Parity is enforced by `scripts/verify-i18n-locales.mjs` (also wired into the web build).

## Locales and UI wiring

Canonical registry: [`modules/i18n/index.ts`](../internal/httpapi/web/modules/i18n/index.ts).

| Concept | Where |
|---------|--------|
| All supported IDs (incl. `pseudo`) | `SUPPORTED_LOCALES` |
| Public picker order | `PUBLIC_LOCALES` (23 locales; speaker-count order) |
| Display labels | `LOCALE_LABELS` |
| Flag SVG paths | `PUBLIC_LOCALE_FLAG_PATHS` → vendored `assets/flags/` |
| In-app RTL `dir` | `ar`, `ur`, `fa` only |
| Locale-changed event | `I18N_LOCALE_CHANGED` (`scrumboy:i18n-locale-changed`) |

`pseudo` is for localhost/QA only (not in the public language picker). Shared picker UI: `locale-select.ts`.

## Runtime expectations

- Resolve copy with `t()` / `hydrateI18n` / `data-i18n-*` at **render or action time**, not at module top level (avoids freezing the startup locale).
- Surfaces that stay open across a locale switch should listen for `I18N_LOCALE_CHANGED` and **remove** listeners on close/teardown.
- Bootstrap embeds `auth.*` and `errors.*` (`BOOTSTRAP_EN_CATALOG`) so sign-in paths work before full catalogs load.
- Preference key: `scrumboy.locale` (localStorage); public picks also mirror to a JS-readable cookie for anonymous landing negotiation.

## API validation errors

- Backend may attach stable `error.details.reason` (`snake_case`).
- Frontend: `apiErrorMessage()` for normal localized mapping; `apiErrorMessageOrRaw()` when unknown/dynamic backend text must stay visible (import/backup diagnostics, etc.).
- Do not invent substring classifiers or localize dynamic warning arrays unless product work explicitly asks.

## Generated landing pages

Anonymous marketing landings are **generated**, not hand-edited:

| Piece | Path |
|-------|------|
| Template | `internal/httpapi/web/landing.template.html` |
| Generator | `scripts/generate-landing.mjs` |
| Outputs | `landing.html` (`en`), `landing.locales/<locale>.html` |

Most `landing.*` strings stay English on non-English pages today; hero subline and multilingual feature card copy are the main localized overrides. Regenerating HTML is required after catalog/template/registry changes (`npm run generate:landing` / `verify:landing`).

SPA module overview (locale flow diagram): [`diagrams/scrumboy_frontend_spa.md`](diagrams/scrumboy_frontend_spa.md).

## Change gate

After i18n catalog, registry, landing, or bootstrap key changes:

```powershell
node internal/httpapi/web/scripts/verify-i18n-locales.mjs
cd internal/httpapi/web; npm.cmd run generate:landing; npm.cmd run verify:landing; npm.cmd test
```

For TypeScript surface changes, also run `npx.cmd tsc --noEmit` in `internal/httpapi/web`. Full contributor rules: [`AGENTS.md`](../AGENTS.md).
